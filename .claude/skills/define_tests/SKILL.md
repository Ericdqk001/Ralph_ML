# /define_tests — Generate Test Suite & PRD for an ML Pipeline Method

## Overview

This skill generates a complete test-driven specification for an ML pipeline method. Tests ARE the specification — they encode scientific correctness, data integrity, and method-specific requirements. The autonomous agent loop (`ralph.sh`) then implements code to pass these tests.

## Usage

```
/define_tests --method <method_name>
```

Example: `/define_tests --method random_forest_classifier`

## Workflow

### 1. Parse the method name

Extract `<method_name>` from the `--method` argument. If not provided, ask the user for it.

The method name determines the directory structure:
- `src/<method_name>/` — implementation code (built by the agent loop)
- `tests/<method_name>/` — test suite (generated by this skill)
- `tasks/<method_name>/` — PRD file (generated by this skill)

### 2. Gather method description

Ask the user to describe their ML method and pipeline. Prompt with:

> **Describe your ML method and what the pipeline should do.**
>
> Include as much detail as you can about:
> - What data does it operate on? (format, source, size)
> - What model/algorithm should be used?
> - What metrics matter? (accuracy, RMSE, AUC, etc.)
> - Any specific preprocessing requirements?
> - Train/validation/test split strategy? (random, stratified, temporal)
> - Any domain-specific constraints or requirements?
>
> If you're not sure about some details, that's fine — I'll suggest sensible defaults.

### 3. Propose pipeline stages

Based on the user's description, propose a set of pipeline stages. If the user doesn't specify, use these defaults:

1. **Data loading** — Load and validate raw data
2. **Preprocessing** — Clean, impute, transform
3. **Feature engineering** — Derive features (if applicable)
4. **Train/val/test split** — Split with appropriate strategy
5. **Model training** — Train with reproducibility
6. **Evaluation & reporting** — Metrics, plots, reports

Present the proposed stages and ask:

> **Here are the pipeline stages I'll generate tests for:**
>
> 1. Data loading (`test_dataloader.py`)
> 2. Preprocessing (`test_preprocessing.py`)
> 3. Feature engineering (`test_features.py`)
> 4. Train/val/test split (`test_split.py`)
> 5. Model training (`test_training.py`)
> 6. Evaluation & reporting (`test_evaluation.py`)
>
> Want to add, remove, or modify any stages?

### 4. Ask clarifying questions

Based on what's still unclear, ask targeted questions. Examples:

- **Data format:** "Is the data in CSV, Parquet, or another format? What are the key columns?"
- **Target variable:** "What is the target/label column? Is this classification or regression?"
- **Metrics:** "Which metrics should the evaluation test check? (e.g., accuracy > 0.8, RMSE < 10)"
- **Split strategy:** "Should the split be random, stratified by target, or temporal?"
- **Reproducibility:** "Any specific random seed preference? (default: 42)"
- **Special requirements:** "Any domain-specific validation rules? (e.g., no negative values in column X)"

Only ask questions that are necessary — don't over-ask. Use sensible defaults where possible.

### 5. Generate the test suite

Create one test file per pipeline stage under `tests/<method_name>/`. Each test file should:

- **Import from `src.<method_name>`** — test the public API that the agent will implement
- **Be self-contained** — each test file can run independently
- **Encode correctness requirements** — shapes, dtypes, value ranges, no NaN where unexpected
- **Enforce ML guardrails** — no data leakage, reproducibility, proper splits
- **Include clear docstrings** — explain what each test verifies and why

#### Test file structure template

```python
"""Tests for <stage_name> stage of <method_name> pipeline."""
import pytest
import numpy as np
import pandas as pd

# Tests import from the implementation module
from src.<method_name>.<module> import <functions_to_test>


class TestStageName:
    """Tests for the <stage_name> stage."""

    def test_basic_functionality(self):
        """Verify the happy path works."""
        ...

    def test_output_shape(self):
        """Verify output dimensions match expectations."""
        ...

    def test_no_data_leakage(self):
        """Verify no information leaks across splits."""
        ...

    def test_reproducibility(self):
        """Verify results are deterministic with same seed."""
        ...

    def test_edge_cases(self):
        """Verify handling of edge cases (empty data, missing values, etc.)."""
        ...
```

#### Key test patterns to include

**Data loading tests:**
- Data loads without error
- Expected columns are present
- Dtypes are correct
- No silent row loss

**Preprocessing tests:**
- Missing values are handled
- Output shapes are consistent
- Fit on train only, transform all splits (leakage check)
- No unexpected NaN introduced

**Split tests:**
- Split ratios are approximately correct
- No overlap between splits
- Stratification preserved (if applicable)
- Reproducible with same seed

**Training tests:**
- Model trains without error
- Loss decreases over epochs (if applicable)
- Model is serializable/saveable
- Reproducible with same seed

**Evaluation tests:**
- Metrics are computed correctly
- Metrics are within expected range (sanity check, not exact values)
- Results are saved to files
- Confusion matrix / diagnostic plots are generated (if applicable)

### 6. Generate the PRD

Create `tasks/<method_name>/prd.json` with one story per test file:

```json
{
  "project": "<Method Name>",
  "branchName": "ralph/<method_name>",
  "description": "<User's description of the method>",
  "userStories": [
    {
      "id": "US-001",
      "title": "<Stage title>",
      "test_file": "tests/<method_name>/test_<stage>.py",
      "priority": 1,
      "passes": false,
      "notes": ""
    }
  ]
}
```

Stories are ordered by priority matching the natural pipeline flow (load -> preprocess -> split -> train -> evaluate).

### 7. Create directory structure

Create the following directories (implementation dirs are empty — the agent loop fills them):

```
src/<method_name>/          # Empty — agent implements here
src/<method_name>/__init__.py
tests/<method_name>/        # Test suite generated above
tests/<method_name>/__init__.py
tests/<method_name>/conftest.py  # Shared fixtures if needed
tasks/<method_name>/        # PRD file
```

Also create a `conftest.py` with shared fixtures (sample data, paths, etc.) if it would reduce duplication across test files.

### 8. Present summary for review

Before writing any files, present a summary:

> **Summary for `<method_name>` pipeline:**
>
> **Stages:** (list)
> **Test files:** (list with brief description of what each tests)
> **PRD:** `tasks/<method_name>/prd.json` with N stories
> **Branch:** `ralph/<method_name>`
>
> **Files to create:**
> - `tests/<method_name>/test_*.py` (N files)
> - `tests/<method_name>/conftest.py`
> - `tests/<method_name>/__init__.py`
> - `src/<method_name>/__init__.py`
> - `tasks/<method_name>/prd.json`
>
> Ready to generate? (Review the test specifications above first)

### 9. Write files and commit

After user approval:
1. Create all directories and files
2. Verify the test files are syntactically valid (they should all fail since no implementation exists yet)
3. Commit with message: `tests: define test suite for <method_name> pipeline`

## Important Notes

- **Tests are the specification** — they must be thorough enough that passing all tests means the pipeline is correct
- **Vertical isolation** — each method is self-contained under its own directory. Duplication across methods is fine.
- **Don't over-constrain** — test for correctness (shapes, no leakage, metrics in range) not exact values (specific loss at epoch 5)
- **Fixtures for test data** — use conftest.py to provide sample/synthetic data that tests can use
- **Imports from `src.<method>`** — tests always import from the implementation module, establishing the API contract
