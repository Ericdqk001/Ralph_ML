# /define_tests — Generate Test Suite & PRD for an ML Pipeline Method

## Overview

This skill generates a complete test-driven specification for an ML pipeline method. Tests ARE the specification — they encode scientific correctness, data integrity, and method-specific requirements. The autonomous agent loop (`ralph.sh`) then implements code to pass these tests.

## Usage

```
/define_tests --method <method_name>
```

Example: `/define_tests --method random_forest_classifier`

## Workflow

### 1. Parse the method name

Extract `<method_name>` from the `--method` argument. If not provided, ask the user for it.

The method name determines the directory structure:
- `src/<method_name>/` — implementation code (built by the agent loop)
- `tests/<method_name>/` — test suite (generated by this skill)
- `tasks/<method_name>/` — PRD file (generated by this skill)

### 2. Stage 1: Pipeline Definition

Ask about each ML pipeline component **one at a time**. Wait for the user's response before moving to the next question. Never assume defaults — gather explicit descriptions for each.

#### Question 1: Data Description

Ask the user to describe their data in detail:

> **Tell me about your data.**
>
> - What is the shape and size of your dataset?
> - What modality is it? (tabular, image, text, time-series, multi-modal)
> - What format is it in and where does it come from?
> - What is the target variable?
> - What is the task type? (classification, regression, ranking, generation, etc.)
>
> Be as specific as possible — this will be used to generate synthetic test data.

Wait for the user's response before proceeding.

#### Question 2: Model Description

Ask the user to specify the model or algorithm:

> **What model or algorithm will you use?**
>
> Please provide the name, and preferably a link to its model page (e.g., Hugging Face, GitHub).

After receiving the answer, **search Hugging Face** (or the web) to find the model:
- If the model is **gated**: inform the user, search official doc and guide the user to set up credentials (e.g., `HF_TOKEN`),
then verify access
- If the model is **public**: confirm it's freely available, no credentials needed

Wait for the user to acknowledge before proceeding.

#### Question 3: Training Regime

> **How will you train the model?**
>
> - Will you use a pre-trained model as-is (inference only)?
> - Or will you fine-tune it?
> - If fine-tuning: which layers to freeze, learning rate, epoch numbers, optimizer, loss function,
and if hyperparameter-tuning, what is the search space?

Wait for the user's response before proceeding.

#### Question 4: Data Preprocessing

> **What preprocessing does your data need?**
>
> - Any data augmentation?
> - Normalization or standardization?
> - Any processing to avoid confounding variables?
> - Which transformations must be fit on train data only (and then applied to val/test)?

Wait for the user's response before proceeding.

#### Question 5: Data Split Strategy

> **How should the data be split?**
>
> - What split ratios?
> - Random vs stratified split? If stratified, which variables to stratify on?
> - Cross-validation? If so, how many folds and what type (K-fold, time-series split, group split)?
> - What factors must be considered to avoid data leakage between sets? (e.g., subject ID, time-based leakage, etc.)

Wait for the user's response before proceeding.

#### Question 6: Evaluation Metrics

> **What metrics matter for this task?**
>
> - Which metrics should be computed? (accuracy, F1, RMSE, AUC, etc.)

Wait for the user's response before proceeding.

### 3. Stage 2: Scientific Rigor Discussion

After gathering all pipeline details, initiate a focused back-and-forth discussion on scientific rigor. This is a conversation, not a checklist — engage with the user's specific pipeline.

#### Topic 1: Data Confounding & Model Bias

Discuss potential confounders in the data, sources of bias, and strategies to avoid shortcut learning. Ask the user what they've already considered and suggest additional concerns specific to their pipeline.

#### Topic 2: Data Leakage

Discuss specific leakage risks for this pipeline: temporal leakage, group leakage, preprocessing leakage. Propose mitigation strategies and get the user's input.

#### Topic 3: Sanity Checks

Discuss what sanity checks should be encoded as tests:
- Baseline comparisons (does the model beat a trivial baseline?)
- Random label tests (does performance drop to chance with shuffled labels?)
- Overfitting-on-small-batch tests (can the model memorize a tiny dataset?)

#### Topic 4: Tailored Test Strategy

Based on all discussion, propose specific tests that verify the user's strategies work as expected. For each proposed test, explain:
- What it verifies
- Suggest if or not the tests can ensure scientific rigor for this pipeline

Iterate on the test strategy until the user explicitly approves.

**Critical:** Do NOT proceed to test generation until the user explicitly says they approve the test strategy.

### 4. Generate the test suite

Create one test file per pipeline stage under `tests/<method_name>/`. Each test file should:

- **Import from `src.<method_name>`** — test the public API that the agent will implement
- **Be self-contained** — each test file can run independently
- **Encode correctness requirements** — shapes, dtypes, value ranges, no NaN where unexpected
- **Enforce ML guardrails** — no data leakage, reproducibility, proper splits
- **Include clear docstrings** — explain what each test verifies and why

#### Test file structure template

```python
"""Tests for <stage_name> stage of <method_name> pipeline."""
import pytest
import numpy as np
import pandas as pd

# Tests import from the implementation module
from src.<method_name>.<module> import <functions_to_test>


class TestStageName:
    """Tests for the <stage_name> stage."""

    def test_basic_functionality(self):
        """Verify the happy path works."""
        ...

    def test_output_shape(self):
        """Verify output dimensions match expectations."""
        ...

    def test_no_data_leakage(self):
        """Verify no information leaks across splits."""
        ...

    def test_reproducibility(self):
        """Verify results are deterministic with same seed."""
        ...

    def test_edge_cases(self):
        """Verify handling of edge cases (empty data, missing values, etc.)."""
        ...
```

#### Key test patterns to include

**Data loading tests:**
- Data loads without error
- Expected columns are present
- Dtypes are correct
- No silent row loss

**Preprocessing tests:**
- Missing values are handled
- Output shapes are consistent
- Fit on train only, transform all splits (leakage check)
- No unexpected NaN introduced

**Split tests:**
- Split ratios are approximately correct
- No overlap between splits
- Stratification preserved (if applicable)
- Reproducible with same seed

**Training tests:**
- Model trains without error
- Loss decreases over epochs (if applicable)
- Model is serializable/saveable
- Reproducible with same seed

**Evaluation tests:**
- Metrics are computed correctly
- Metrics are within expected range (sanity check, not exact values)
- Metrics results are saved/returned in a structured format

### 5. Generate the PRD

Create `tasks/<method_name>/prd.json` with one story per test file:

```json
{
  "project": "<Method Name>",
  "branchName": "ralph/<method_name>",
  "description": "<User's description of the method>",
  "userStories": [
    {
      "id": "US-001",
      "title": "<Stage title>",
      "test_file": "tests/<method_name>/test_<stage>.py",
      "implementation_order": 1,
      "passes": false,
      "notes": ""
    }
  ]
}
```

Stories are ordered by implementation_order matching the natural pipeline flow (load -> preprocess -> split -> train -> evaluate).

### 6. Create directory structure

Create the following directories (implementation dirs are empty — the agent loop fills them):

```
src/<method_name>/          # Empty — agent implements here
src/<method_name>/__init__.py
tests/<method_name>/        # Test suite generated above
tests/<method_name>/__init__.py
tests/<method_name>/conftest.py  # Shared fixtures if needed
tasks/<method_name>/        # PRD file
```

Also create a `conftest.py` with shared fixtures (sample data, paths, etc.) if it would reduce duplication across test files.

### 7. Present summary

Before writing any files, present a summary:

> **Summary for `<method_name>` pipeline:**
>
> **Pipeline stages** (as discussed):
> (list each stage with a one-line description)
>
> **Test files:**
> (list with brief description of what each tests)
>
> **PRD:** `tasks/<method_name>/prd.json` with N stories
> **Branch:** `ralph/<method_name>`
>
> **Files to create:**
> - `tests/<method_name>/test_*.py` (N files)
> - `tests/<method_name>/conftest.py`
> - `tests/<method_name>/__init__.py`
> - `src/<method_name>/__init__.py`
> - `tasks/<method_name>/prd.json`

### 8. Write files and commit

After user approval:
1. Create all directories and files
2. Verify the test files are syntactically valid (they should all fail since no implementation exists yet)
3. Commit with message: `tests: define test suite for <method_name> pipeline`

## Important Notes

- **Tests are the specification** — they must be thorough enough that passing all tests means the pipeline is correct
- **Vertical isolation** — each method is self-contained under its own directory. Duplication across methods is fine.
- **Never provide defaults** — always ask the scientist for explicit answers
- **Do not proceed to test generation until the user explicitly approves** the test strategy
- **Fixtures for test data** — use conftest.py to provide sample/synthetic data that tests can use
- **Imports from `src.<method>`** — tests always import from the implementation module, establishing the API contract
